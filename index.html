    The Web Design Process
    Typography - Typeface

    Decide on the main typefaces

    2-3 type faces maximum

    (more typefaces = worse performance)

    “Typeface” = “font-family” in CSS

    The Web Design Process
    Typography - Typeface

    Decide on a font-stack - the list of fonts you would like to see, in order of preference, e.g.:

    Raleway, Helvetica Neue, Helvetica, Lucida Sans, Arial, sans-serif

    This means “use Raleway, but if we don't have that font then use Helvetica Neue, but if we don't have that font then use Helvetica...” etc.

    The Web Design Process
    Typography - Typeface

    Provide the typefaces for non-standard fonts

    Font-providers like Google fonts, typekit, and fonts.com* provide links for loading fonts

    *other providers are available

    “Non-standard” = fonts that aren't already installed on users' devices. Note that different platforms (Windows, Mac OS, iOS, Android, Linux, etc.) may have different fonts installed by default.

    The Web Design Process
    Typography - Typeface

    Linked fonts quick and easy to implement (happy developers) and fast to load (happy users, happy Google)

    If you can't find a link to the font, then provide files

    The Web Design Process
    Typography - Typeface

    Font files should be in WOFF format. Not OTF. Not TTF. WOFF

    If you don't have the files in WOFF format, then provide a link

    If you don't have a link or WOFF files then choose a different typeface!

    Further reading:

    • Wikipedia article on the Web Open Font Format (WOFF)
    • Current browser support for the WOFF file format

    Typography - Typeface

    The Web Design Process
    Typography - Base Font Size

    Everything on the page inherits this font size

    The Web Design Process
    Typography - Base Font Size

    All spacing should be relative to this font size

    The Web Design Process
    Typography - Base Font Size

    Browsers set the base font size at 16px by default

    The base font size may be changed based on the screen width

    e.g. 14px on small screens, 16px on larger screens, 20px on very large screens…

    The Web Design Process
    Typography - Other Font Sizes

    Font sizes may be specified in:

    • px – for the base font size
    • rem – relative to the base font size
    • em – relative to the font size of the parent element
    • % – relative to the font size of the parent element

    The Web Design Process
    Typography - Other Font Sizes

    Apart from the base font size, you should never use px

    Why fixed sizes are bad

    The Web Design Process
    Typography - Other Font Sizes

    A good strategy:
    • base font size in  px
    • each component has a rem font size
    • elements inside each component may have an em font size

    That means everything can scale easily by changing the base font size, and all sizes remain in proportion

    The Web Design Process
    Typography - Other Font Sizes

    All sizes, especially margins and padding, should be specified in relative units (em)

    Relative = Scalable

    Why fixed sizes are bad

    Typography - Baseline Grid

    Web Design Pitfalls
    Large High-Fidelity Mockups

    Remember, we're designing systems, not pages

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Large file size
    Mockups are huge
    • It takes a long time to upload/download them
    • They take up a lot of disk space
    • The contain a lot of redundant data - sections of pages (components) may be unnecessarily included in multiple mockups

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Time-consuming
    • They take a long time to make
    • While you're making them the developer is idle
    • Updating a mockup is slower than updating code
    • It takes the developer a long time to extract useful information
    • time = money; mockups make the process more expensive

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    No UX

    Static mockups make it difficult, if not impossible, to convey:

    • Animation
    • Active/Focus/Hover states
    • Form validation states & notifications
    • Interactivity
    • User action sequences

    CSS makes it easy!

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Unrealistic
    • The resulting website will never look the same as the mockup on all devices
    • A mockup is just a picture of a webpage
    • If you're presenting page mockups to clients, you're being dishonest

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Untestable
    • Mockups are just pictures of websites; you can't load them on different devices and test them or interact with them
    • You can't gather any useful feedback from users or clients

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Static

    Mockups may look great with the content you paste in, but what about on the live product, with real, dynamic content?

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Fixed-width

    This makes them unsuitable for web design

    How big should your canvas be?

    Separate “mobile”, “tablet”, and “desktop” mockups just means more time & money without addressing the problem

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Discourages modularity
    • It forces you to design a whole page, all at once
    • Creating small modules in high-fidelity is an unnecessary step and defeats the purpose of a high-fidelity mockup - to convey the look & feel

    Web Design Pitfalls
    Large High-Fidelity Mockups
    Why Full-Page Mockups Are Bad

    Developers are not Photoshop experts
    • Developers need to focus their time on coding, not on copying & pasting from mockups, or measuring sizes and trying to calculate the relative units
    • It's not their area of expertise
    • Your design deliverables are supposed to help developers

    Web Design Pitfalls
    Fixed Sizes

    NEVER apply fixed sizes to anything that may contain variable content!!!

    Variable content requires variable sized containers

    Web Design Pitfalls
    Fixed Sizes

    NEVER specify the size of anything in pixels, unless you are 100% that the size should not vary across device/platform!

    Sizes should be specified in relative units, relative to the font size

    Web Design Pitfalls
    Fixed Sizes

    Sizes should be consistent! (e.g. all margins should be a specific multiple of the base font size)
